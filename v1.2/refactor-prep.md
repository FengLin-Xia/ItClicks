# V1.1 → V1.2 重构前准备

本文档在不动代码的前提下，梳理当前资产、V1.2 目标对照、以及重构时的保留/移除/新增清单与建议实施顺序。实际写代码前可据此做任务拆解与排期。

**实施约定**：重构代码全部落在 **v1.2/** 目录下，不修改 v1.1 目录。v1.2 含独立 backend 与 frontend，可单独运行。

## 一、当前代码资产清单（V1.1）

### 1.1 仓库与目录


| 路径                        | 说明                         |
| ------------------------- | -------------------------- |
| `v1.1/frontend/`          | 前端静态页                      |
| `v1.1/backend/`           | Node + Express 后端          |
| `v1.1/nginx.conf`         | Nginx 配置                   |
| `v1.1/docker-compose.yml` | 部署编排                       |
| `v1.1/backend/logs/`      | SQLite 日志（generation_logs） |


### 1.2 前端页面


| 文件            | 作用                                    | 与 V1.2 关系                          |
| ------------- | ------------------------------------- | ---------------------------------- |
| `index.html`  | 关系原型生成页：展示 seed、换一换 / 代一代，跳画布         | **逻辑替换**：改为刷流入口或刷流首屏               |
| `canvas.html` | 画布页：关系状态可视化 + 三操作（压一层/换视角/暴露一点）+ 建议卡片 | **大幅改或弃用**：V1.2 无画布，改为「卡片 + 自定义生成」 |


### 1.3 后端 API

**约定：全部现有接口保留。** 不删、不合并，仅在现有基础上新增接口（如 feed、customize 等）。


| 路由                | 方法   | 作用             | 与 V1.2 关系                                  |
| ----------------- | ---- | -------------- | ------------------------------------------ |
| `/api/generate`   | POST | mode=seed      | relation_op，返回 text + relation_state       |
| `/api/log-action` | POST | 记录用户行为（采纳/放弃等） | **保留**。用于 V1.2 指标（刷流停留、自定义点击/完成、分享、二次生成等）。 |
| `/health`         | GET  | 健康检查           | **保留**。                                    |


### 1.4 后端核心逻辑（server.js）


| 模块                               | 说明                                            | 与 V1.2 关系                                                            |
| -------------------------------- | --------------------------------------------- | -------------------------------------------------------------------- |
| 限流                               | 内存 Map，60s 内 30 次                             | **保留**，可沿用或按「刷流 vs 生成」分开限流                                           |
| SQLite generation_logs           | 记录每次生成的请求/响应/重试                               | **保留**，便于分析；可增加字段区分 feed/custom                                      |
| Session 状态                       | lastSeedMode、lastSeedForm、seedHistory 等       | **部分废弃**：画布态不再需要；若做「预生成池」可保留类似去重/分布逻辑                                |
| TENSION_MODES / EXPRESSION_FORMS | 张力模式 × 表达形式，用于 seed 多样性                       | **可复用**：预生成内容池的「结构均匀分布」可基于此类维度                                       |
| Writer + Critic 双阶段              | 先生成再批判、重试                                     | **保留思路**：自定义生成同样需要质量约束，Critic 可改为「语感/张力/长度」                          |
| Seed / Relation Op Prompts       | 多套 system prompt（deepen/perspective/reveal 等） | **重构**：Seed 类 prompt 用于预生成或首条；Relation Op 收敛为「自定义生成」单流程 + 结构抽取 + RAG |


### 1.5 数据流（V1.1）

```
index: 调用 /api/generate(mode=seed) → 展示 → 代一代 → localStorage 写入 session → 跳 canvas
canvas: 读 session → 展示关系可视化 + 画布 → 三按钮调用 /api/generate(mode=relation_op, op=...) → 展示建议卡片 → 采纳/放弃 → /api/log-action
```

V1.2 目标流：

```
刷流页: 预生成内容池（可接口或静态）→ 无限滑动卡片 → 每张卡片底部「试试生成你的版本」
       → 进入自定义：用户输入 1–3 句 → 结构抽取 → Prompt + RAG → 80–120 字 → 可导出卡片
```

## 二、V1.2 目标与差异对照


| 维度     | V1.1                         | V1.2                        |
| ------ | ---------------------------- | --------------------------- |
| 入口     | 单条关系原型（seed）                 | 刷流（预生成内容池 + 无限滑动）           |
| 主交互    | 画布 + 三关系操作                   | 卡片浏览 + 单次「自定义生成」            |
| 生成触发   | 每次点按钮即调用 API                 | 刷流多为预生成/缓存；仅「自定义」实时调 API    |
| 输出形态   | 1–2 句、关系状态可视化                | 80–120 字代餐文本、可导出卡片图         |
| 状态模型   | polarity / initiative / tags | 弱化或仅后台用；前台以「语感 + 张力」为主      |
| Prompt | 多 op 多 prompt                | 统一「自定义生成」流程：结构抽取 + 模板 + RAG |
| 指标     | seed 转化、relation_op 次数、采纳率   | 刷流停留、自定义点击/完成、分享、二次生成       |

## 三、保留 / 移除 / 新增 清单

### 3.1 保留（可沿用或小改）

- **后端**：Express 框架、限流、健康检查、Dotenv、DeepSeek 调用方式、日志表（可扩展字段）。
- **前端**：静态部署方式、请求 `/api/`* 的写法、基础样式思路（可做卡片化）。
- **Prompt 思路**：高压缩、不解释、不绑定设定、张力控制；Critic 可改为「代餐语感 + 长度」校验。
- **TENSION_MODES / EXPRESSION_FORMS**：可用于预生成池的分布与去重，避免内容过窄。

### 3.2 移除或大幅收缩

- **画布页**：canvas.html 的「关系状态可视化 + 三操作 + 建议卡片」整体不再作为主流程；若需可保留为隐藏/实验页。
- **关系状态可视化**：极性条、主动权、Tags 的 UI 与强依赖可移除或仅保留在后台/调试。
- **relation_op 三种 op**：接口保留；前端主流程不再暴露为多个按钮，改为「自定义生成」一条链路；画布页可保留为隐藏入口继续用现有 API。
- **Session 中的画布态**：lastSeedMode、lastSeedForm、seedHistory 等与画布强绑定的可删或仅服务「预生成池」去重。

### 3.3 新增

- **刷流层**
  - 预生成内容池：数据来源（定时任务调现有 seed 接口写入 DB/文件，或首次加载时批量生成并缓存）。
  - 前端：无限滑动、卡片式列表、每张卡片可分享图片（含文案）。
  - 卡片底部 CTA：「试试生成你的版本」跳转或展开自定义。
- **自定义层**
  - 输入：1–3 句自由描述或简单关系设定。
  - 后端：结构抽取（可先规则/关键词，后续可接模型）→ Prompt 模板 → RAG 注入圈层语感 → 80–120 字生成。
  - 输出：高张力代餐文本 + 可导出卡片图（前端 canvas 或服务端生成图片）。
- **RAG**：圈层语感/例句库（文档或向量），在自定义生成时注入，保证风格统一。
- **埋点/日志**：自定义点击、完成、分享、二次生成，便于算核心指标（见 PRD 第 9 节）。

## 四、刷流页改动讨论

本节只讨论刷流页（Layer 1 入口）的改动，不涉及自定义层与后端实现细节。便于先对齐「从哪改、长什么样、怎么交互」。

### 4.A 定位与目标

- **页面的作用**：入口 + 情绪唤起；制造「差一点就说到心坎上」的轻度不满足，引导点「试试生成你的版本」。
- **成功标准**：刷流停留时间、自定义点击率（PRD 核心指标）；体验上「即时加载、不能卡」。

### 4.B 从哪一页改


| 方案               | 做法                                                               | 优点                       | 缺点                       |
| ---------------- | ---------------------------------------------------------------- | ------------------------ | ------------------------ |
| **A. 替代 index**  | 把现有 index.html（关系原型单条 + 换一换/代一代）整体替换为刷流页，原入口可改为 `/seed.html` 或隐藏 | 用户首次打开即刷流，路径短；一个入口       | 需处理「从刷流去画布」的入口（若保留画布为隐藏） |
| **B. 新页面**       | 新建 feed.html，index 保留为关系原型页或改为跳转选择（去刷流 / 去画布）                    | 不动现有 index/canvas 逻辑，风险小 | 多一层选择或需要默认跳 feed         |
| **C. index 内切换** | index 上增加 Tab/切换：「刷流」 / 「单条原型」，默认展示刷流                            | 单页内兼顾两种入口                | 交互略重，MVP 可能不必            |


**定稿**：采用 **A（替代 index）**。现有 index.html 整体替换为刷流页；原「关系原型单条 + 换一换/代一代」迁至 `seed.html`，从刷流页底部或隐藏入口可进。

### 4.C 信息结构（一屏里有什么）

- **单条卡片**  
  - 主内容：一段预生成文案（与现有 seed 同质：高压缩、关系向、不解释），长度可略放宽到 2–4 句以便「情绪命中」。  
  - 可选：不展示 polarity/initiative/tags（弱化关系状态，突出情绪与代入感）。
- **列表**  
  - 竖向无限滑动，一屏约 1–2 张卡片（视卡片高度而定），避免一次加载过多导致卡顿。
- **每张卡片上的操作**  
  - 主 CTA：**「试试生成你的版本」**（必选）。  
  - 次要：**分享/保存为图片**（可 MVP 先做「复制文案」或简单导出图）。

### 4.D 数据从哪来（与现有 API 的关系）

- **方案 1：纯前端调现有接口**  
  - 刷流页加载时或滑动到底时，前端直接调 `POST /api/generate`（mode=seed），一次请求一条，拼进列表。  
  - 优点：零后端改动，立刻能跑。  
  - 缺点：每次滑动都可能等一次生成，易卡、成本高、难做「随机+结构均匀分布」。
- **方案 2：新接口 GET /api/feed**  
  - 后端从「预生成内容池」读数据，返回列表（如 cursor + limit）；内容池由定时任务或后台用现有 seed 接口预填。  
  - 优点：刷流即时加载、不卡、成本可控、可做分布与去重。  
  - 缺点：需要后端做内容池与 feed 接口。
- **方案 3：混合**  
  - 首屏 N 条走 `GET /api/feed`（或静态 JSON），后续滑到一定位置再按需调 `POST /api/generate`（mode=seed）追加并缓存到前端列表。

**定稿**：采用 **方案 2**。刷流数据来自新接口 `GET /api/feed`（cursor + limit），后端从预生成内容池读取；内容池由定时/后台任务调用现有 `POST /api/generate`（mode=seed）填充。

### 4.E 交互细节（可进一步定稿）

- **首屏**：进入即展示 1–2 张卡片；若用 feed 接口则直接渲染，若用 generate 则首条可 loading 后出文案。  
- **无限滚动**：触底（或接近触底）加载更多；加载中可显示占位或小 loading，避免整页卡住。  
- **「试试生成你的版本」**  
  - **定稿**：点击后 **跳新页**，进入「自定义输入 1–3 句」的流程。当前卡片的文案随跳转带给新页（如 query 或 state），新页可只读展示或隐藏传给后端，便于「基于这条的改写/延展」。
- **分享**：MVP 可「复制文案」；若做导出图，可与自定义结果页的「导出卡片」共用一套样式（文案 + 固定排版）。

### 4.F 样式与体验

- **卡片**：白底/浅底、圆角、适度阴影，与现有 index 的 seed-display 气质接近即可，便于复用风格。  
- **不能卡**：首屏渲染优先；请求能缓存的就缓存（同一 session 内相同 cursor 不重复要）；若用 generate 逐条拉，考虑「预取下一条」在空闲时发。  
- **无推荐算法**：列表顺序随机或按时间/均匀分布即可，不做「猜你喜欢」。

### 4.G 与画布、单条原型的关系（在保留全部接口的前提下）

- 刷流页 **不** 提供「代一代进画布」的主入口，避免路径分叉。  
- 若保留画布：可在刷流页底部或设置里放「玩关系画布」链到 `canvas.html`；画布仍用现有 `POST /api/generate`（mode=relation_op）与 `POST /api/log-action`。  
- 若保留单条关系原型页：可作 `seed.html`，从刷流页隐藏入口进入；单条页仍用 `POST /api/generate`（mode=seed）与「代一代」跳画布。

### 4.H 刷流页规格说明（定稿）

实施刷流页时按以下规格执行。


| 项         | 结论                                                                                                  |
| --------- | --------------------------------------------------------------------------------------------------- |
| **页面**    | 替代现有 `index.html`，主入口即刷流；原关系原型单条页迁至 `seed.html`，可选保留隐藏入口。                                           |
| **数据**    | 使用 **方案 2**：前端调 `GET /api/feed?cursor=&limit=` 获取预生成卡片列表；内容池由后端用现有 seed 逻辑预填并存储。                    |
| **自定义入口** | 卡片底部「试试生成你的版本」点击后 **跳新页**（如 `customize.html?ref=...` 或通过 state 传递当前卡片 id/文案）；新页负责输入 1–3 句并走自定义生成流程。 |
| **首屏**    | 进入即请求 feed，渲染 1–2 张卡片，即时加载不卡。                                                                       |
| **加载更多**  | 触底请求下一批（cursor + limit），追加列表，可带 loading 占位。                                                         |
| **卡片内容**  | 仅展示预生成文案（2–4 句），不展示 polarity/initiative/tags。                                                       |
| **分享**    | MVP 可为「复制文案」或简单导出图，与自定义结果页导出样式可复用。                                                                  |

## 五、技术重构清单（实施时可勾选）

### 5.1 后端

- **保留全部现有接口**：`/api/generate`、`/api/log-action`、`/health` 不改动、不删除。
- **内容池**：设计预生成内容的存储（表或文件），以及生成/刷新策略（定时 or 按需）；可内部调用现有 `POST /api/generate`（mode=seed）填充。
- **刷流接口（新增）**：如 `GET /api/feed?cursor=&limit=` 返回预生成卡片列表（可带简单去重与分布）。
- **自定义生成接口（新增）**：如 `POST /api/customize`，入参 `{ input: "1–3 句" }`，出参 `{ text: "80–120 字" }`；内部走结构抽取 → Prompt → RAG → Writer(+ Critic)。
- **结构抽取**：第一版可用规则/关键词；若有需要再上小模型。
- **RAG**：圈层语感库的录入与检索接口，并在自定义生成 pipeline 中调用。
- **限流**：刷流读接口可宽松；自定义生成严格限流（含未登录时的「基础次数」逻辑预留）。
- **日志**：generation_logs 增加 type=feed|custom、custom_input 等字段；log-action 增加事件类型（custom_click、custom_done、share、regenerate）。

### 5.2 前端

- **刷流页**：新页面或替代现有 index：无限滑动、卡片组件、分享图（文案 + 简单排版）。
- **自定义入口**：卡片底部「试试生成你的版本」→ **跳新页**（带当前卡片文案/ref），新页输入框 1–3 句。
- **自定义结果页/层**：展示 80–120 字结果 + 「导出卡片」按钮（生成图并下载）。
- **移除/折叠**：关系原型单条页与画布页从主流程移除或改为隐藏入口。

### 5.3 Prompt 与生成

- **预生成用 Prompt**：在现有 seed 类 prompt 上微调，保证「情绪命中但不完全贴合」、结构均匀（可结合 TENSION_MODES/EXPRESSION_FORMS）。
- **自定义生成 Prompt**：统一模板，约束长度 80–120 字、不直白、高张力；接入 RAG 注入语感。
- **Critic**：从「关系状态」改为「代餐语感 + 张力 + 长度」检查，不通过则重试或返回提示。

### 5.4 运维与配置

- **环境**：保留 .env、Docker/nginx 部署方式；如需 RAG，增加向量服务或简单文件检索配置。
- **文档**：README 更新为 V1.2 结构（刷流 + 自定义）、新 API 说明、核心指标含义。

## 六、建议实施顺序

1. **文档与约定**
  - 确认 PRD V1.2 与本文档一致。  
  - 约定「预生成内容池」的 MVP 形态（例如：先 100 条静态/半静态，还是全部接口生成后落库）。
2. **后端：自定义生成闭环**
  - 先实现 `POST /api/customize`（不含 RAG 也可），从用户输入到 80–120 字输出，并写日志。  
  - 再加结构抽取与 RAG、Critic。
3. **前端：自定义流程**
  - 在现有某页或新页做「输入 1–3 句 → 调 /api/customize → 展示结果 → 导出卡片图」。  
  - 导出卡片可先简单（文案 + 固定样式），再优化排版。
4. **后端：刷流与内容池**
  - 定预生成存储与刷新策略，实现 `GET /api/feed`。  
  - 用现有 seed 逻辑或改造后的 seed 逻辑批量生成并写入内容池。
5. **前端：刷流页**
  - 无限滑动 + 卡片列表，卡片底部 CTA 跳转/展开自定义。  
  - 分享图可先复用「文案 + 固定样式」，与自定义结果页的导出一致。
6. **埋点与指标**
  - 在关键动线埋点（刷流停留、自定义点击、完成、分享、二次生成），与 log-action 和日志表对齐，便于看 PRD 第 9 节指标。
7. **收尾**
  - 从主导航移除或隐藏画布/关系原型单页；更新 README 与部署说明。

## 七、风险与依赖

- **内容池规模**：过小会重复，需要预生成策略 + 去重/分布（TENSION_MODES 等）在 MVP 就考虑。  
- **RAG 数据**：圈层语感库需要一定量的优质例句或段落，否则「注入语感」收益有限。  
- **导出卡片**：若在前端用 canvas 生成图，需注意跨端一致性；若后端生成，需选图库（如 node-canvas / sharp）与字体。  
- **成本**：刷流尽量用预生成减少实时调用；自定义生成限流 + 基础次数，为后续付费留口。
